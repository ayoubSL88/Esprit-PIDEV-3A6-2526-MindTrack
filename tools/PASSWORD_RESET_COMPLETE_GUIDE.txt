# Password Reset - Complete Implementation Guide

## Overview

MindTrack’s password reset flow is already implemented as a **service-driven flow** in the JavaFX app. It is not a REST API. Instead, the UI controller calls services that:
1. generate a 6-digit reset code,
2. store a hashed copy in MySQL,
3. email the code to the user,
4. verify the code,
5. update the password,
6. mark the token as used.

---

## Main Files Involved

### 1) `src/main/java/controllers/ForgotPasswordController.java`
Handles the UI flow:
- request reset code
- verify reset code
- set new password
- return to login screen

### 2) `src/main/java/services/PasswordResetService.java`
Contains the reset business logic:
- `requestReset(email)`
- `verifyCode(email, code)`
- `resetPassword(email, code, newPasswordHash)`

### 3) `src/main/java/services/PassRestEmailService.java`
Sends the actual email with the reset code.

### 4) `src/main/java/utils/SmtpConfig.java`
Loads SMTP settings from environment variables or properties.

### 5) `src/main/java/services/UtilisateurService.java`
Used to find the user by email and update the password hash.

### 6) `src/main/java/utils/PasswordHasher.java`
Used to hash the reset code and the new password.

### 7) Database table: `password_reset_tokens`
Stores reset tokens, attempts, expiry, and usage state.

---

## How the Flow Works

### Step 1: User enters email + captcha
The controller validates:
- email is not empty
- email format is valid
- captcha is correct

If valid, it calls:
```java
resetService.requestReset(email)
```

### Step 2: Service generates and stores reset code
`PasswordResetService.requestReset()`:
- finds the user by email
- checks rate limits
- generates a random 6-digit code
- hashes the code using `PasswordHasher.hash(code)`
- stores the hash in `password_reset_tokens`
- sends the plain code by email

### Step 3: User receives email
`PassRestEmailService.sendPasswordResetCode()` sends:
- subject: `MindTrack password reset code`
- body: code + expiry message

### Step 4: User enters the code
The controller calls:
```java
resetService.verifyCode(email, code)
```

This checks:
- user exists
- token exists
- token is not used
- token is not expired
- attempt count is not exceeded
- code matches the stored hash

### Step 5: User enters new password
The controller hashes the new password and calls:
```java
resetService.resetPassword(email, code, PasswordHasher.hash(newPassword))
```

If valid:
- password is updated
- token is marked as used
- user is redirected to login

## Current Password Reset Service Code Structure

### `PasswordResetService`

#### Main methods
- `requestReset(String email)`
- `verifyCode(String email, String code)`
- `resetPassword(String email, String code, String newPasswordHash)`

#### Security features
- 6-digit reset code
- 10-minute expiry
- maximum 5 attempts
- 60-second cooldown between requests
- 5 reset requests per hour
- token marked as used after success
- password updated inside a DB transaction

#### Verification statuses
```java
OK
INVALID
EXPIRED
TOO_MANY_ATTEMPTS
NOT_FOUND
```

## Email Service

### `PassRestEmailService`
This service sends the reset code through SMTP.

It requires:
- SMTP host
- SMTP port
- SMTP username
- SMTP password
- SMTP from address

If SMTP is missing, it throws a `MessagingException`.

If Gmail authentication fails, it throws:
```java
SMTP authentication failed. Check Gmail App Password.
```

## Database Table

### `password_reset_tokens`
The service expects a table like this:

```sql
CREATE TABLE password_reset_tokens (
    id INT AUTO_INCREMENT PRIMARY KEY,
    user_id INT NOT NULL,
    code_hash VARCHAR(255) NOT NULL,
    expires_at TIMESTAMP NOT NULL,
    used TINYINT(1) DEFAULT 0,
    attempts INT DEFAULT 0,
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    FOREIGN KEY (user_id) REFERENCES utilisateur(idU)
);
```

### Stored fields
- `user_id` - linked user
- `code_hash` - hashed reset code
- `expires_at` - expiry time
- `used` - whether token was consumed
- `attempts` - invalid attempts count
- `created_at` - request time

## Important Behavior

### Token selection
The service always uses the **latest active token** for the user.

### Rate limiting
A user cannot spam reset requests because the service checks:
- last request cooldown: 60 seconds
- hourly maximum: 5 requests

### Password update safety
Password update and token marking are done inside a transaction so both succeed or both fail together.

---

## Edge Cases

1. **Unknown email**
   - The service returns silently to avoid leaking whether an account exists.

2. **Expired code**
   - Token is marked used and `EXPIRED` is returned.

3. **Too many attempts**
   - Token is marked used and `TOO_MANY_ATTEMPTS` is returned.

4. **Wrong code**
   - Attempts counter increments.

5. **Repeated reset requests**
   - Cooldown + hourly limit prevent abuse.

6. **SMTP failure**
   - Email sending throws an exception that the controller shows to the user.

7. **Database failure**
   - Controller displays a generic database error.

## Configuration

### SMTP settings
The app uses `SmtpConfig` for:
- host
- port
- username
- password
- from address

Make sure these are configured in your environment or `smtp.properties`.

### Password hashing
The reset code is stored hashed, not in plain text.
The new password is also hashed before saving.
